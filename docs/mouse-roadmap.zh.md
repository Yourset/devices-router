# 鼠标能力后续规划

## 现状

Devices Router 目前不是完整鼠标共享工具。现有鼠标能力只做一件事：通过两端鼠标活动推断键盘目标应该在主电脑还是副电脑。

这条能力的边界很重要：

- Logitech Flow 继续负责真实鼠标跨屏移动。
- Devices Router 只根据鼠标活动切换键盘目标。
- 游戏模式下默认保守，避免自动切换打断主电脑游戏或窗口操作。

## 为什么不要直接做完整鼠标迁移

完整鼠标迁移比键盘转发更危险，主要风险是：

- 鼠标位置是连续状态，不像按键是离散事件，丢一帧或重复一帧都会有明显体感。
- 需要处理多屏幕、DPI 缩放、窗口边界、坐标映射、加速度、滚轮、拖拽、按键状态。
- 鼠标拖拽跨机器容易造成“鼠标按钮卡住”或目标窗口误操作。
- 和 Logitech Flow 同时工作时，两个系统都在控制鼠标，容易抢控制权。

因此，鼠标迁移不应该直接混进当前键盘链路里做。它应该作为独立阶段设计、独立开关、独立日志。

## 推荐路线

### 阶段 1：巩固当前鼠标跟随

目标：让“鼠标活动切换键盘目标”稳定、可解释、可关闭。

要做：

- 日志写清楚是谁触发切换：主电脑鼠标活动、副电脑鼠标活动、手动按钮、快捷键。
- 设置页明确说明游戏模式会关闭自动鼠标切换。
- 网络诊断里显示最近一次鼠标活动来源和时间。
- 保留三档灵敏度，但避免暴露过多毫秒参数。

### 阶段 2：鼠标意图检测，而不是鼠标转发

目标：更聪明地判断用户是不是“真的想切过去”。

可选策略：

- 连续活动阈值：副电脑鼠标活动持续一小段时间才切键盘。
- 冷却时间：刚切回主电脑后短时间内不被副电脑抢回。
- 前台应用保护：主电脑检测到游戏模式或全屏应用时不自动切走。
- 用户可见日志：每次不切换也要能解释原因，例如“游戏模式开启，已忽略副电脑鼠标活动”。

### 阶段 3：实验性鼠标事件转发

目标：在明确关闭 Logitech Flow 或特定测试模式下，尝试鼠标点击、滚轮、移动转发。

必须满足：

- 默认关闭。
- 独立实验开关，不和键盘跟随混在一起。
- 禁止游戏模式下启用。
- 先做滚轮和点击，再考虑移动。
- 移动必须先支持坐标归一化和屏幕边界保护。
- 所有鼠标 down 都必须有自动 key-up / button-up 兜底，避免卡住。

## 不建议做的事

- 不建议在当前键盘修复尚未完全稳定时直接做完整鼠标共享。
- 不建议和 Logitech Flow 同时抢鼠标位置。
- 不建议默认开启鼠标事件注入。
- 不建议把鼠标转发和键盘转发写进同一条协议处理分支，后续会难以排查。

## 下一步建议

短期继续修界面和诊断，不急着做完整鼠标迁移。

优先级：

1. 修 UI 稳定性：避免频繁重绘影响输入框、按钮和日志。
2. 完善日志：每条自动切换都要说明触发来源和是否被游戏模式拦截。
3. 增加“最近输入链路状态”：主电脑是否转发、副电脑是否注入、目标窗口是否管理员权限。
4. 再评估是否进入鼠标意图检测阶段。
