# Flow Keyboard Bridge 路线图

本文整理下一阶段的产品迭代方向，面向当前的 Tauri/Rust 桌面版主线。

## 现在的基线

当前版本已经具备这些核心能力：

- 主电脑 / 副电脑模式切换
- 自动发现主电脑，或手动填写主电脑地址
- 鼠标跟随的基础判定
- 日志清空、复制、导出
- 自动更新与开机自启
- 基础主题和常用状态展示

下一阶段的目标，不是把它做成完整 KVM，而是把“日常使用里最烦的几个点”补齐。

## 迭代原则

- 先让状态看得懂，再让配置好管理
- 先让排障更快，再扩展更复杂的场景
- 继续保持“键盘桥”为主线，鼠标转发只作为后续探索
- 所有新增能力都要能被普通用户直接感知，不做只对开发者有意义的功能

## 路线图

| 阶段 | 目标 | 重点交付 |
| --- | --- | --- |
| P0 | UI 状态补齐 | 选中态、启动中态、运行中态更明确 |
| P1 | 用户设置收拢 | 把分散的偏好统一到设置区 |
| P2 | 日志与网络诊断 | 让用户自己就能定位大部分连接问题 |
| P3 | 鼠标跟随三档 | 把跟随手感做成可选预设 |
| P4 | 游戏模式 | 让游戏场景更稳，不抢输入 |
| P5 | 未来鼠标转发 | 作为后续探索项，不放在当前主线前面 |

## P0. UI 状态补齐

这一步的目标是减少“我现在到底有没有点到”的不确定感。

- 主电脑 / 副电脑按钮增加更清楚的选中态
- 启动、切换、连接过程增加启动中态或处理中态
- 当前模式、当前目标、当前连接状态一眼可见
- 在切换过程中避免重复点击造成误操作
- 日志区的状态按钮和页面主按钮保持一致的视觉反馈

## P1. 用户设置收拢

现在已有的偏好项比较分散，下一阶段要把它们整理成更像“用户设置”的东西。

- 启动方式
  - 开机自启
  - 记住上次模式
  - 最小化到托盘
- 网络与连接
  - 自动发现开关
  - 主电脑地址
  - 端口信息展示
- 视觉偏好
  - 主题切换
- 输入偏好
  - 鼠标跟随档位
  - 游戏模式开关

目标不是堆更多开关，而是把用户真正会改的内容放在同一处。

## P2. 日志与网络诊断

日志区是最重要的排障入口，下一阶段要把它从“能看”提升到“好用”。

### 日志能力

- 复制日志
- 导出日志
- 暂停滚动与恢复滚动
- 重复日志折叠
  - 例如连续重复的心跳、发现、连接提示合并显示
  - 需要时可以展开查看完整原文

### 网络诊断

- 主电脑可达性检测
- 发现端口检测
- 键盘端口检测
- 更新服务检测
- 当前主机地址提示
- 诊断结果直接给出“哪里断了”

这一阶段的目标是：用户不用先发截图给开发者，也能自己判断是地址错、端口没放行，还是自动发现没生效。

## P3. 鼠标跟随三档

把鼠标跟随从“一个默认值”变成“可理解的三档”。

- 稳定
  - 更保守，减少误切换
  - 适合日常办公和长时间挂机
- 平衡
  - 默认推荐档位
  - 在稳定性和响应速度之间取中间值
- 灵敏
  - 更快响应鼠标活动
  - 适合追求切换速度的用户

这里的重点不是暴露更多技术参数，而是让用户用“体验标签”选，而不是用毫秒数选。

## P4. 游戏模式

游戏模式的目标很明确：当用户在主电脑上打游戏时，不要因为误判或频繁切换影响体验。

- 提供明确的游戏模式开关
- 游戏模式下优先保留本地输入稳定性
- 降低误切换概率
- 在界面上清楚提示当前是否处于游戏模式
- 后续再考虑是否支持按场景自动启用

这部分要优先保证“稳”，不要为了激进响应牺牲可预测性。

## P5. 未来鼠标转发

鼠标转发是后续探索方向，不是当前迭代的默认范围。

可先评估这些问题：

- 转发哪些内容
  - 指针移动
  - 鼠标按键
  - 滚轮
- 哪些场景适合转发
  - 日常办公
  - 远程输入补位
  - 是否和游戏模式冲突
- 哪些边界必须保留
  - 安全窗口
  - 管理员窗口
  - 游戏窗口

如果后续要做，建议先做研究和原型验证，再决定是否纳入主线。

## 不变的边界

- 继续以 Windows-to-Windows 为主
- 不把项目扩成完整 KVM
- 不去模拟 Logitech Flow 的私有协议
- 鼠标仍然主要由现有 Flow 负责，键盘桥负责补齐缺口

## 建议的推进顺序

1. 先做 UI 选中态 / 启动中态
2. 再收拢用户设置
3. 再补日志与网络诊断
4. 然后上线鼠标跟随三档
5. 接着做游戏模式
6. 最后再评估鼠标转发
